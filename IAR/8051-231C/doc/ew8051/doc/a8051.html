  • Upgrading from V2.XX
    • Reserved Symbols
      You can get a listing of all symbols via the -xDI command line option.
    • Directives
              - PUBLIC and EXTERN do not allow a type specifier.
              - LOCSYM accepted but not executed.
              - LSTWID accepted but not executed.
              - LSTFOR accepted but not executed.
              - TITL accepted but not executed.
              - PTITL accepted but not executed.
              - STITL accepted but not executed.
              - PSTITL accepted but not executed.
              - The LST... directives do not remember earlier LST... directives. I.e. they are not of the counting type.
              - Cross-reference table printed differently.
              - DS 0 does not force alignment.
    • The syntax for floating point numbers does not allow a leading double quote. I.e. you cannot write DD "1.0.
    • The PUBLIC and EXTERN directives do not take type information.

      Support for Micro-series Assembler Macros in IAR assembler 
    • For compatibility with A8051, V2 the old macro syntax is also supported. Please use the new syntax when you write new code.
    • Note that there are two types of macro definitions:

      1.  Intel/Motorola macros. These are defined:
      "macname MACRO  args". Please see the manual for details.

      2. IAR Micro-Series macros. These are defined:
      "MACRO  %macname".

      Note that you cannot have a label in the beginning of that line. You can use both types of macros in a program, but you cannot use Micro-series operators in Intel/Motorola macros. Intel/Motorola macros allow \A through \Z as argument references, but they are not valid for Micro-series macros (some of them are used as operators).

      For more information on the IAR Micro-Series macros, please see the A8051 V2 manual.

  • Operators in A8051
    Please use the command line option -u to get compatibility with V2.xx.
  • Source format in A8051
    Directives and mnemonics are not allowed in the label field, i.e. they need a space, tab, or label to their left.

Known problems

  • The following code sequence generates an internal error:
                    macro %PUSHA
                    PUSH A
                    endmac
             MY_DATA:
             D_START EQU MY_DATA
             MY_CODE:
                    %PUSHA
                    IF (D_START < 128)
                    NOP
                    ENDIF
                    END
    [8051A0024]
  • The assembler cannot handle symbols with bit addresses in segments other than bit segments.
    [8051A0023]

    • The file asm.xcl defined the wrong cpu.

      [8051A0022]
    • The option -u did not work for AND, OR, or XOR.
      [8051A0021]
    • The assembler generated "Error[43]: Illegal effective address" for SETB and CLR where the address should be legal.
      [8051A0020]
    • The ORL and ANL did not support the logical complement "/" for some bit-addressed bits.
      [8051A0019]
    •
  • 3.20A
    • On the SUN platform, the SETB instruction would generate:
      Error[43]: Illegal effective
      [8051A0016]

  • 3.11A
    • In an expression where the multiplication operand (*) is evaluated together with operands of the same precedence, the multipication operand will be evaluated after the other operands, not left-to-right.
      [8051A0015]
    • Relative bits were assigned the wrong addresses if more than one source file was included in the project. The bits were assigned absolute addresses.
      [8051A0014]
    • The predefined symbol P had the wrong address (0xB0.5) associated with it. The address is changed to 0xD0.0.
      [8051A0013]

    • It was not possible to check the length of a definition field.
